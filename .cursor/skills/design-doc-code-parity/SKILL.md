# 设计文档与代码严格等效（Design-doc parity）

当仓库里存在与源码**同目录**、可版本管理的 **`.design.md`**（或可执行逻辑规格段落）时，把它当作**改代码的契约**：实现行为、顺序、阈值、分支与文档**严格等效**；文档若滞后，要么更新文档，要么改代码，**二者不得长期矛盾**。

本技能管 **「规格 ↔ 实现」差分、文档撰写/拆分、末尾计划执行、验收与提交节奏**；通用 AS 写法见 **write-angelscript**；跑测与失败迭代见 **angelscript-tdd-agent-iteration**；中文原子提交见 **git-commit-zh**；ManteumTower 等多仓路径与 Git 根见 **git-local-p4-workflow**。蓝图/索引大范围搜索见 **blueprint-snapshot-search**；断言编辑器内蓝图事实见 **soft-ue-cli-ue-bridge**。

---

## 0. 「执行计划」与用户指令（语义）

会话里常出现两种「计划」，**默认语义不同**：

| 用户说法 | **默认**应执行的对象 |
|----------|----------------------|
| **「执行计划」** / **「执行设计文档中的计划」** / **「落实末尾计划」** / **「按文档计划改」**（无路径） | **`*.design.md`** 末尾 **`## 待执行计划`** 中 **Task 注册表** 里状态**非「已实现」**的条目：逐条改 **等效代码与测试**，**逐条**改状态；**仅当全部已实现**时整节 **清空**（§5.3），并把真值写回正文。 |
| **「按 Cursor 计划执行」** / **「执行 `…/xxx.plan.md`」** | 用户给出的 **`.plan.md`** 文件中的步骤（可能与 design 无关或为会话辅助）。 |

**禁止**在仅听到「执行计划」时，**自动**改去执行 **`.cursor/plans/*.plan.md`** 或本轮对话里助手自拟的待办，除非用户明确要那份文件。

**执行前**：对目标 **`*.design.md`** 做 §2.1 的**合并阅读**，标出 **`## 待执行计划`** 里仍与代码/测试不一致的条款，再按 §5.1 引用门禁与 §5.2 分阶段落地。

**执行时（禁止「只改 design」）**：**执行计划** = 按条目修改 **等效源码与测试**；**禁止**仅通过重写 `.design.md`、删除「待执行计划」或改正文措辞来「交差」而 **`.as` 未变**（见 §5.0）。

### 0.1 讨论设计 vs 执行计划（**默认不同步改代码**）

基于 **「先迭代等效设计文档，再按 `## 待执行计划` 落代码」** 的节奏：

| 用户在做的事（示例） | **默认允许改** | **默认不要改** |
|----------------------|----------------|----------------|
| 「问设计」「这套方案」「帮改设计文档」「补 `## 待执行计划`」「拆 `*-deferred.design.md`」 | **`*.design.md`**（及文内链接的拆分文档、仓库根 `plan-*.md` 若适用） | **等效 `*.as` / 测试** |
| **「执行计划」** / **「执行设计文档里的计划」** / **「按待执行计划改代码」** / **「对齐代码」** / **「实现」** / 明确「改 `Foo.as`」 | **源码与测试**，再按 §5.3 收敛 design | — |

**禁止**：

- 用户**仅**在讨论、撰写或修改 **设计**（未说执行计划、未说实现/对齐代码）时，助手**同时**提交对 **`.as`** 的修改——会跳过「规格定稿 → 再执行」的迭代，**架空**基于等效 design 的工作流。
- 把「帮我想想设计」自动当成「顺便把代码也改了」。

**若需要落代码但用户未开口**：只更新/建议 **`.design.md`** 与 **`## 待执行计划`**（Diff 级 Task），并**一句**说明：下一步需用户说 **「执行计划」**（或等价指令）后再改 `.as`。

**与 §2.3a 的关系**：§2.3a 适用于用户**已**要求**落地实现**（删变量、改行为、执行 Task 等）；**不**适用于纯设计迭代阶段——该阶段**默认不动** `.as`（见上表）。

## 1. 何时启用

- 用户**只**讨论/撰写/修改 **`*.design.md`**（含 `## 待执行计划` 草稿）：按 **§0.1** — **默认只动文档**，不自动改 `.as`。
- 用户要求：按 `*.design.md` **改代码**、与 design **严格等效**、**对齐代码**、消除 §「规格与实现差异」等（含 **执行计划**）— 再动 `.as` 与测试。
- **撰写或更新**与代码等效的设计文档、从 `.as` 反写规格、**拆分** `*.movement-deferred.design.md` 类迁出文档（若未要求执行，仍以文档为主）。
- 用户要求：按文档末尾 **`## 待执行计划`** **分阶段改代码/测试**，再收敛正文（§5.3）。
- 存在**拆分文档**时（例如主文档只写「动画前数据」，导航/MoveTo 在 `*.movement-deferred.design.md`）：先读文首**范围说明**，避免把已迁出规格误当成「当前仍须实现的代码」。

## 2. 标准步骤（顺序固定）

### 2.1 读规格、定范围

1. 打开目标 **`*.design.md`**（及链接到的拆分文档、仓库根计划如 `plan-*.md` 若文内引用）。
2. **若将执行文档末尾 `## 待执行计划`**：必须先**合并阅读**——主 `.design.md` + 全部 **拆分文档** + 文内链接的 **`plan-*.md`**——再分阶段对照代码；禁止只读主文档就删成员或改 Tick。
3. 标出：**哪些章节对应当前 `.as` 必须满足**；哪些标注为「已迁出 / 待阶段 X / 仅现状描述」——后者**不**作为单独删代码的依据，除非同步执行对应计划（避免双轨行为缺失）。

### 2.2 对照实现

1. 打开对应源码（如 `UMTAnimInstance.as`、`BPC_AnimComponent.as`）。
2. 按文档结构逐项对照：**初始化顺序**、**每帧/ Tick 子序列**、**公开函数公式与常量**、**默认曲线名/属性表**、**调试与节流**。
3. 记录差异表：| 文档条款 | 代码位置 | 一致？ | 处理（改代码 / 改文档 / 收窄注释） |

### 2.3 分类处理

| 类型 | 动作 |
|------|------|
| 行为与文档不一致 | **改代码**（或先改测试再改代码），使与文档一致 |
| 文档落后于已合并实现 | **改 `.design.md`**，使与代码一致（若用户以文档为权威则反过来改代码） |
| 文件头注释与实现不一致，但 design 已写明「当前实现」 | **收窄注释**或改实现；与 **angelscript-tdd-agent-iteration** 跑测后提交 |
| 文档写明「规格与实现差异（二选一）」 | 与用户或 § 说明选定权威侧，**同步**另一侧（代码 + design 同一提交或连续原子提交） |

### 2.3a 实现变更：**先代码，后规格**（禁止只抠 design）

**前提**：用户已要求 **落地实现**（含 **执行计划**、明确「删变量/改行为/改 `*.as`」），**不是**仅处于 **§0.1** 的设计讨论阶段。

用户要求涉及 **运行时行为、成员变量、调试点、打印、Tick 路径**（例如「删掉无效的 debug 打印变量」「去掉某 `UPROPERTY`」「改某函数逻辑」）且**未**说「只改文档 / 只更新 design」时：

| 步骤 | 要求 |
|------|------|
| **1** | **先**改 **等效源码**（`*.as` 等）及**必要时的测试**；使实现与意图一致。 |
| **2** | **再**改 **`.design.md` 规格正文**（属性表、调试节、公式、数据契约），使与 **改后代码** **等效**（删行、收窄或标注「已移除」——与代码一致）。 |

**禁止**：

- **只改 `.design.md`**，从规格正文/变量表/公式里 **删除** 某变量或字段名，而 **`.as` 未删或未改**——会造成 **文档无、代码仍有** 或 **用户要删代码却删了文档** 的错位，**破坏 design ↔ code 等效**。
- **只改 `.design.md`**，从 **`## 待执行计划`** 里抹掉某 Task 或把 Task 标「已实现」，而 **对应 Diff 未在代码中落地**。
- 把「用户要删代码里的 debug 变量」理解成「把设计文档里的变量描述删掉即可」。

**待执行计划** 的维护仍须遵守：**已完成 Task → 按 §4.2/§5.3 清理或推进状态**；**新动作 → 新增 Task**，且每条 **Diff 等效**、可映射到具体文件与符号；**禁止**用「改规格正文」代替 **Task 里对代码的变更**。

### 2.4 TDD 与门禁

1. **Git 工作目录**：游戏工程在 **`Engine\ManteumTower`**（或项目约定的 `.uproject` 仓库根），勿在错误层级提交（见 **git-local-p4-workflow**）。
2. **基线**：在含 `ManteumTower.uproject` 的目录执行 `.\Scripts\RunAngelscriptTests.ps1`，确认 **PASS** 后再改行为。
3. **改行为**：先调整/新增 `Script/Test/` 或模块内测试断言（若项目已有约定），再改实现，再跑到 **退出码 0**（细节见 **angelscript-tdd-agent-iteration**）。
4. **仅注释/文档**：仍建议跑一遍 AS 测试作为**回归门禁**（编译与脚本完整性）。

### 2.5 原子化验收提交

1. **一次提交一个可验收点**（例如：仅 UMT 调试路径、仅 BPC 某函数、仅文档对齐）。
2. **Subject 中文 UTF-8**，可检索、避免模糊词（见 **git-commit-zh**）。
3. **Kit 与项目分仓**：改 Kit 文档与改 ManteumTower 脚本**不要**混在同一 commit。

## 3. 与拆分文档、主计划配合

- **主 `.design.md` 收窄**后，若代码仍调用已「迁出」章节中的函数（例如 Tick 仍调 `Movement()`），以**主 design 文首说明**为准：可能标注为**现状** + **后续瘦身见 `plan-*.md`**。此时**不**擅自删除调用，除非同时落地替代方案（AI 控制器、虚拟 MoveTo 等）。
- 迁出文档（如 `*.movement-deferred.design.md`）用于**对照与未来重构**，不是「立刻删实现」的单独指令。

## 4. 从代码撰写或更新 `.design.md`（Authoring）

1. **通读**目标 `.as`（及相关类），按 **§6 附录** 结构起草或**增量更新**现有 `.design.md`。
2. **不发明**代码中不存在的分支、阈值或 API；若仅存在于注释，在文档中标注 **「规格与实现差异」** 或推动收窄注释（与 §2.3 一致）。
3. **拆分前**：先明确**主文档服务的数据链**（例如「仅 Anim 前数据 / 与 `UMTAnimInstance` 的契约」），再把无关切面迁到 **`<Base>.<concern>-deferred.design.md`**，主文档文首写清范围与链接。
4. 复杂类可保留 **变更记录**表（日期/摘要/可选 commit 指针）；**勿**在正文长期保留已执行完毕的整段「计划」正文（见 §5.3）。

### 4.1 末尾 **`## 待执行计划`**（Task 注册表 + Diff 正文）

每次改设计、需要留下「尚未落地到代码」的条目时：

- **标题**：最后一节使用 **独立标题** **`## 待执行计划`**（可选用 **`## 待执行计划（Diff）`**）。**不要**使用 `## 8`、`## 9` 等与正文规格 **`## 0.` … `## N.`** 混排的连续章节号；**计划块**单独成节、**不沿用**前文小节序号。

- **小节结构（固定顺序）**  
  1. **Task 注册表（放在本节最前）**：用表格或列表 **集中列出全部 Task**（建议 id：`T1`、`T2`… 或稳定短名），每一行**必须**有 **状态** 字段，取值限定为：**待执行** / **进行中** / **已实现**。  
  2. **逐 Task 正文**：在注册表**之后**，按 id **分小节**（如 `### T1 …`）书写该 Task 的 **Diff 等效**说明（目标文件、符号、`-`/`+`、前置门禁等）。**禁止**只有注册表没有正文，或只有正文没有注册表。

- **状态规则**  
  - **禁止**只把整节标成「状态：已实现」或一句话概括完成——**不够**；必须 **每个 Task 在注册表里各自更新状态**。  
  - 执行过程中：完成某 Task 的代码/测试后，**只**把该 Task 标为 **已实现**；其余保持 **待执行** 或 **进行中**。  
  - **未全部完成时**：后续新增/修改计划时，**在现有注册表与正文上增量编辑**（补 Task、改 Diff、推进状态），**不要**另起一节重复堆积「旧已实现」列表；**新增 Task** 须与 **期望代码 Diff** 一一对应，**禁止**把「改规格措辞」写进 Task 冒充实现变更。

- **Diff 正文**：每条须 **可机械执行**。至少包含：**目标文件**、**符号或位置**、**具体变更**。**禁止**单独使用「优化」「对齐」「重构」等无锚点措辞。

- **与 §2.3a 一致**：Task 若描述「删除某变量 / 某调试成员」，**实现侧**必须改 **`*.as`**；design 侧只在代码改完后 **同步**规格，**禁止**仅在 Markdown 里删掉变量名交差。

- **可附**：前置条件（如引用门禁）写在对应 Task 正文内。

### 4.2 全已实现后的 **清空**（不堆积）

- 当 **Task 注册表中每一个** Task 均为 **已实现**，且对应行为已写回 **前面规格正文**（数据契约、Tick 顺序等）后：**整段删除 `## 待执行计划` 及其全部内容**（含注册表与各 Task 小节），**不要**保留「状态：已实现」的遗迹占用篇幅。  
- **若仅部分已实现**：**保留**本节，只更新注册表状态与未完成 Task 的正文；**不得**因「想看起来干净」而删掉未完成的 Task。

## 5. 执行文档末尾 **`## 待执行计划`**

当 `.design.md` 含 **待执行**或**分阶段**计划（如删成员、镜像 BPC 字段、删占位 API）时：

### 5.0 执行语义

- **执行计划** = 按 **`## 待执行计划`** 内 **Task 注册表** 逐条推进：对状态为 **待执行/进行中** 的 Task 修改 **等效源码与测试**；每完成一条，在注册表中把该 Task 标为 **已实现**（**禁止**只把整节标成已实现而子 Task 仍混为待执行）。  
- 全部 Task **已实现** 且正文已合并真值后：按 §5.3 **整节删除** `## 待执行计划`，**禁止**长期堆积「已全部已实现」的注册表。  
- **禁止**：**只改 `.design.md`**（改状态、删字）而不改 **`.as`**；**禁止**把 **规格正文** 改成与未改代码自洽。

### 5.1 前置：引用门禁

- 在删 `UPROPERTY`、删 `UFUNCTION`、改 Tick 顺序前：对 **AS** `grep` + **`.soft-ue-index`**（或项目 Blueprint 文本索引）检索 **Get/绑定**；需要断言「当前编辑器内」蓝图事实时，在 **soft-ue-cli** `check-setup` 成功后用 **query-blueprint** 等（**soft-ue-cli-ue-bridge**）。
- **无引用或已解除绑定**后再删符号，避免加载断线。

### 5.2 分阶段实现

- 按 **Task id**（`T1`→`T2`… 或文档内约定顺序）执行：每完成一个 Task 的 **代码与测试**，**立即**在注册表更新该 Task 为 **已实现**，再进入下一 Task；每步 **TDD 全绿** 再推进。  
- **设计文档侧**：每步只改 **注册表状态** + 必要时微调该 Task 正文（若实现与 Diff 有出入，先以代码为准再收紧文档）；**不要**跳过逐条状态、只写一句「本节已完成」。
- 文档要求 **「非必要不简化」** 的数学（如 Locomotion **滞回、阈值、死区、`UpdateDesiredLocomotionState` 全链路**）：默认**不**合并、不「顺手优化」；仅删除文档明确允许删除的占位 API（如历史上的无状态 `CalculateSpeedRange()`），且**不**连带改掉内部 `GetSpeedRangeBoundsFromLocomotionState` 实现。

### 5.3 执行后：逐条状态 → 全清

- **部分完成**：仅更新 **Task 注册表** 与各 Task 状态；**保留** `## 待执行计划` 整节，便于继续执行剩余 Task。  
- **全部完成（清空条件）**：当且仅当 **每一个** Task 在注册表中均为 **已实现**，且承诺行为已写回 **前面各节正文**（数据契约、Tick 顺序、公式等）后：从 `.design.md`（及拆分文档若也有对称小节）中 **整节删除 `## 待执行计划`**（注册表 + 各 Task 正文一并删），**不**保留「状态：已实现」的堆积块。  
- **合并真值**：删除整节前，确认正文已吸收计划中的行为；历史以 **Git** 为准，必要时在文首 **变更记录** 增一行（日期 + 摘要 + commit），**不**把已删计划正文贴回文档。  
- **AnimBP 等人工项**：未完成时对应 Task 保持 **待执行** 或拆成独立 Task；勿用整节占位代替逐条 Task。

## 6. 附录：`*.design.md` 模板与维护规则

1. **命名与位置**：与源码同目录，`<ClassOrModule>.design.md`；跨切面迁出：`<Base>.<concern>-deferred.design.md`（例：`*.movement-deferred.design.md`）。文首写明 **范围、与主文档关系、当前 `.as` 是否仍包含迁出逻辑**。
2. **文首元信息**：一两句说明「可据此改代码」、权威侧（改代码后是否必须同步本文档）。
3. **正文编号**：`§0` 约束；`§1` 初始化；`§2` 每帧或 Tick **子序列（顺序固定）**；公式节写清 **常量、分支、早退**；调试或屏显单独成章，或用 **「规格与实现差异」**。
4. **数据契约表**（跨类时）：列 **写入方 / 只读方**（如 BPC → `LocomotionState` → UMT 仅镜像），避免 AnimInstance **重复** BPC 速度带数学。
5. **`## 待执行计划`**：置于文档**最后**；**最前**为 **Task 注册表**（每 Task 必有 **待执行/进行中/已实现**）；**其后**逐 Task 写 **Diff 级**正文（§4.1）。**全部 Task 已实现** 后 **整节删除**（§4.2、§5.3），**禁止**堆积「已实现」遗迹；未完成则只增量改注册表与剩余 Task。
6. **硬约束短节**（按需）：如滞回/死区非必要不简化；§x.x 仅只读镜像、不重复某函数——便于在执行阶段拒绝跑题「优化」。
