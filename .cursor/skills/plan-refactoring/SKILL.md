# 制定重构计划

当用户要求**制定重构计划**、**评估项目状态并给出重构路径**、或**输出可执行的重构/迁移计划**时，按本技能执行：**理解要求 → 评估项目状态 → 制定计划 → 输出计划**。所有路径以**项目根**（代码与资源所在工作区根）为基准；若涉及 Kit，则 Kit 根为含 `.cursor/skills` 的仓库。

---

## 1. 根据要求（理解需求与目标）

- **明确范围**：用户要重构的是整仓、某模块、还是某类资产（如「所有武器相关」「Blueprint→AS」）？
- **明确目标**：目标形态是什么（例如：逻辑迁到 AS、蓝图仅做资产/子类；或某模块拆成多个子模块）？
- **约束与优先级**：有无时间/风险约束、必须先动的模块、不能动的接口？
- 若用户未说清，在回复中**先列出假设**（范围、目标、优先级），再基于假设给出计划；或简短追问关键一两项。

---

## 2. 评估项目状态

在制定计划前，对当前项目做**最小必要**的评估（避免大而全的扫描，聚焦与本次重构相关的部分）。

### 2.1 依赖与边界

- **谁依赖谁**：列出与重构范围相关的依赖（模块/类/蓝图/脚本之间的引用、事件、接口）。
- **入口与出口**：哪些是外部调用入口（如蓝图事件、C++ 接口、AS 暴露的 UFUNCTION）；哪些是必须保持兼容的契约（函数签名、事件名、资产路径）。
- **读编辑器内蓝图**：用 **soft-ue-cli**（`check-setup` 成功后 `query-blueprint`、`query-blueprint-graph` 等，见技能 **soft-ue-cli-ue-bridge**）理清依赖与图结构；**桥不可达时先** **ue-editor-launch** 或 **`content/dev/scripts/Start-UnrealEditor.ps1`** 恢复交互式 UE 与 Bridge，再查询。必要时结合 **Git** 历史（技能 **git-local-p4-workflow**）。

### 2.2 范围与粒度

- **可独立验证的单元**：将范围拆成可单独编译、运行、测试的子域（如「武器」「角色」「动画」）。
- **已有实现**：若为迁移（如 BP→AS），列出已有 AS/C++ 实现与对应蓝图，标出「已对齐 / 未对齐 / 仅部分实现」。

### 2.3 基线与风险

- **行为基线**：当前「正确行为」如何被定义或验证（自动化测试、关键用例、快照）？若无，在计划中标注「需先建立基线」。
- **回滚**：是否容易回滚（版本控制、分支、保留旧蓝图/旧类一段时间）。

---

## 3. 制定重构计划

基于「要求」与「项目状态」，产出一份**可执行、分阶段**的计划。

### 3.1 策略选择

- **Strangler Fig**：新功能用目标形态（如 AS），旧实现逐步被替换；双轨运行一段时间。
- **Facade/Adapter**：新实现（如 AS）承担核心逻辑，通过接口/委托供旧调用方使用；旧侧先做薄壳。
- **按子系统替换**：以领域为单位（如武器、角色、动画）整块迁移，而不是单文件/单蓝图散点替换。
- **一次性替换**：仅当范围小、依赖少、风险可控时采用。

### 3.2 阶段与优先级

- 将计划拆成**阶段**（如阶段 0：准备与基线；阶段 1：子系统 A；阶段 2：子系统 B；阶段 3：收尾）。
- 每个阶段内给出**优先级顺序**（先做哪一块、后做哪一块），并注明**依赖关系**（若 B 依赖 A，则 A 在前）。
- 每阶段包含：**范围**、**输入/前置条件**、**步骤摘要**、**验收方式**、**回滚或补救**（若适用）。

### 3.3 步骤粒度

- 单步应**可在一个会话或一次提交内完成**，且完成后可编译/可运行/可测。
- 明确「**读什么、改什么、验证什么**」：例如「用 soft-ue-cli 查蓝图 A → 在 AS 中实现事件 X → 在编辑器中把 BP_XXX 父类改为 AS 类 → 跑用例 Y」。

### 3.4 与 Kit/规则协作（若适用）

- 若项目遵循 **AS 优先**（rule angelscript-primary-cpp-fallback）：计划中写逻辑的迁移目标为 AS；蓝图仅保留资产或薄壳时，在计划中写明。
- **读图与改行为**：理解逻辑用 soft-ue-cli / 编辑器；**修改**仅在 AS、C++ 或编辑器内改 `.uasset`，不存在「改导出 JSON 修蓝图」的路径。
- 若需写 AS：计划中可引用技能 **write-angelscript**；若需测试驱动迭代，引用 **angelscript-tdd-agent-iteration**；若为 Blueprint→AS 迁移，可参考本技能下的 **reference.md**（Blueprint→AS 专项路径）。

## 4. 输出计划

将上述内容整理为**结构化、可直接执行或交接**的文档。

### 4.1 输出格式

建议包含（可按需增删）：

1. **目标与范围**：一句话目标；本次计划覆盖的模块/资产类型。
2. **假设与约束**：依赖的假设（如「仅考虑武器与角色」）；不可动接口、必须保留的兼容性。
3. **项目状态摘要**：依赖关系要点、已有实现与缺口、基线/测试现状。
4. **策略**：采用的总体策略（如 Strangler Fig + 按子系统替换）。
5. **阶段列表**：每阶段含名称、范围、优先级、步骤摘要、验收方式。
6. **风险与缓解**：主要风险（如无自动化测试、某接口调用方未知）；建议的缓解（如先加冒烟测试、先迁移只被一处调用的模块）。
7. **后续可沉淀**：建议将本次经验写入知识库的条目（如 `content/knowledge/NN-refactoring-bp-to-as.md`），可引用技能 **summarize-to-knowledge**。

### 4.2 交付方式

- 在回复中直接给出**计划正文**（Markdown 结构化列表/表格）。
- 若用户需要持久化：建议写入项目或 Kit 的 `content/plan.md`、`content/design/` 或 `content/knowledge/`，并在回复中注明路径，便于后续 @ 引用。

- [ ] **输出**包含目标、假设、状态摘要、策略、阶段、风险与缓解；必要时写出文件路径供沉淀
- [ ] 若为 Blueprint→AS：已参考本技能 **reference.md**，且计划中遵守 soft-ue-cli 读图、AS 优先
