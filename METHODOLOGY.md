# 研发过程 / 设计-测试-验证-迭代 / 迭代计划 / 知识库 — 通用方法论

本文档定义「设计→测试→验证→迭代」的文档结构、编写约定及与 AI 的协作方式。适用于采用本 Kit 的研发型项目（如游戏、引擎、工具链等）。

**本 Kit 分为核心泛化层与具体内容层**：核心层（METHODOLOGY、.cursor/rules、templates）**高度泛化、可迁移、自我解释**，不依赖 content/ 或项目内的具体路径、文件名或技术栈；可单独迭代与迁移至其他项目。**具体内容层（content/）** 为当前项目专属，由 Agent 或团队根据项目生成，所有文档与知识库均在 **本 Kit 根目录的 content/** 下，项目路径下只放实际功能代码与资源。

---

## 版本控制与仓库约定

- **Kit 为独立 Git 仓库**：本 Kit 以独立仓库维护；克隆/拉取仅包含核心泛化层。仓库内 **.gitignore 忽略 content/**，故所有知识库与项目内容（content/）不纳入 Kit 版本控制。
- **核心层自包含**：METHODOLOGY、.cursor/rules、templates **不引用 content/ 或项目内的具体路径、文件名、技术栈**，仅描述 content 下目录角色与用法；因此核心层可独立阅读、迁移，不依赖任一项目的 content。
- **项目内容不影响核心层**：对 content/ 的增删改仅作用于 content/；不修改、不依赖 METHODOLOGY、.cursor/rules、templates。同理，对核心层的修改在 Kit 仓库内提交，与 content/ 无关；两方迭代互不干扰。
- **核心层迭代**：对 METHODOLOGY、rules、templates 的修改在 Kit 仓库内提交与打 tag，与 content/ 分离，便于复用与迁移。

## 文档结构（Kit 根下的 content/）

以下路径均在**本 Kit 根目录的 content/** 下：

| 文档/目录 | 路径 | 用途 |
|----------|------|------|
| 迭代计划 | `content/plan.md` | 阶段/里程碑、任务拆解、状态、验证与迭代记录 |
| 设计文档 | `content/design/` | 按功能或模块的设计文档（目标、方案、接口、技术栈相关设计要点等） |
| 开发文档 | `content/dev/` | **实际开发内容**：实现说明、编辑器工具、导出流程、配置与操作步骤等 |
| 知识库 | `content/knowledge/` | **通用知识**：概念、决策记录、易错点、跨模块的约定与参考 |
| 代码与 API 参考 | `content/reference/` | 示例代码、API 文档、schema、导出 JSON 等，主要供 AI/工具引用 |

### knowledge 与 dev 的边界

- **knowledge/**：更**通用**，偏「为什么、注意什么、选型与约定」。适合：领域概念与约定、项目级决策记录、易错与排错备忘、与 reference 的对应关系说明等。不放入具体编辑器操作、某工具/插件的使用步骤、导出路径与命令等「怎么做」的细节。
- **dev/**：更**针对实际开发**，偏「怎么做、用哪些工具、输出在哪」。适合：编辑器/工具使用说明、导出流程与目录约定、插件/模块集成与编译步骤、脚本与自动化用法等。若某经验同时具备「通用教训」与「具体操作」，可把操作细节写在 dev，在 knowledge 的决策/易错类文档中留一条简短引用。

- 设计阶段产出写入 `content/design/`；开发与实现说明、工具与导出流程写入 `content/dev/`；概念、决策、易错点写入 `content/knowledge/`。
- 代码片段、接口定义、API 说明、数据格式与 schema 等放在 `content/reference/`，便于 AI 与工具直接引用。
- 完成某阶段或某任务后，在 `content/plan.md` 中更新状态、验证结果与迭代记录，并视情况同步 design/dev/knowledge。

## 初始化步骤（供 Agent 执行）

初始化只做一次，由**项目 Agent** 按以下原则执行，无需脚本：

1. **在项目根下添加 rule**  
   在项目根的 `.cursor/rules/` 下新增 `use-ue-dev-kit.mdc`，注明：本项目使用本 Kit；文档与知识库在 Kit 的 content/；按 METHODOLOGY 工作。

2. **在 Kit 根下生成 content/**  
   根据项目需要在 Kit 根下创建 **content/**，可包含：`plan.md`、`design/`、`dev/`、`knowledge/`、`reference/` 等；**结构由 Agent 按项目决定**，可参考本 Kit 的 templates/ 作为建议结构。

## 迭代计划文件（plan.md）结构

- **阶段/里程碑表**：表格列：阶段、名称、状态、最近更新。状态可选：`未开始` | `进行中` | `已完成`。
- **任务拆解表**：表格列：顺序、任务名、阶段、优先级、状态、说明。状态同上。
- **阶段详情**：每个阶段一节，含：状态、最近进展、下一步、验证结果、迭代记录。
- 与 design/dev 配合：各阶段可对应 design 中的设计文档与 dev 中的实现说明；更新阶段状态时可一并更新相关设计/开发文档。

## 设计文档（content/design/）

- 目录下按功能或模块放置设计文档。
- 每篇可含：目标、方案、接口约定、与项目技术栈相关的设计要点（模块划分、资源与管线等）。
- 含 `README.md` 说明目录结构与各文件简介，并指向 plan.md。
- 设计文档约定可同步至团队知识库或协作平台；同步方式由项目自定。

---

## 开发文档（content/dev/）

- 目录下放置**实际开发内容**：实现说明、关键接口、配置说明、编辑器/工具使用步骤、导出流程与目录约定、插件/模块集成与编译步骤、脚本与自动化用法等。
- 可按子目录组织（例如按工具、管线、模块划分），具体结构由项目在 content 内自定。
- 含 `README.md` 说明目录结构与各文件简介，并指向 plan.md 与 design/、knowledge/。
- 开发文档约定可同步至团队知识库或协作平台；同步方式由项目自定。

---

## 知识库（content/knowledge/）

- 目录下为**通用**知识沉淀：概念、决策记录、易错点、与 reference 的对应关系（按技术域或主题分文件，具体文件名与结构由项目在 content 内自定）。
- 建议项目在 knowledge 下维护**架构总览**文档（如 `01-project-architecture-overview.md`），用于描述项目根目录结构、C++ 插件与 AS 模块、蓝图分布与核心资产、以及与其他 knowledge 文档的引用关系；便于 AI 与新人快速建立项目骨架。具体文件名与结构由项目自定，可参考 templates 中的说明。
- 含 `README.md` 说明目录结构与各文件内容简介，并指向 plan、design、dev。
- **不放入**：具体工具的操作步骤、导出路径与命令、某插件/工具的使用手册等，这些归 `content/dev/`。
- 使用方式：研发过程中将关键概念、决策、链接、易错点整理到对应文件；在 Cursor 中 @ 引用可让 AI 基于本地知识库回答。**约定**：知识库可作为团队知识库来源同步至协作平台；同步方式由项目自行决定，本 Kit 不提供同步脚本。

---

## 代码与 API 参考（content/reference/）

- 目录下放置示例代码、API 文档、schema、配置片段等**以机器/工具消费为主**的参考内容，非以人读为主的叙述性文档。
- 供 AI 与自动化工具引用，减少幻觉并贴合项目实现。可含：接口签名、示例代码片段、数据格式说明等。
- 含 `README.md` 说明目录结构及各文件用途，便于 AI 选择引用。

---

## 与团队知识库/协作平台的同步（约定）

- `content/knowledge/`（以及视项目需要的 `content/design/`、`content/dev/`）可作为团队知识库来源，**约定**同步到团队使用的知识库或协作平台，便于检索与协作。
- 实现方式由项目自行决定，本 Kit 仅做约定与文档结构统一，不提供同步脚本或对具体平台的依赖。

## 设计/开发文档的编写约定

以下约定适用于**设计文档、开发文档**及新增/修订相关文档时：

1. **须包含可执行的「信息」**：每个条目尽量包括：
   - **概念**：相关术语、设计决策与原因；
   - **操作**：具体实现步骤或配置方式（含路径、确认方式）；
   - **验证**：怎样算完成、如何自测或验收。
2. **命令与步骤须逐条解释**：凡出现命令或多步操作，对每一条单独说明「做什么、为什么」。
3. **新增或修订设计/开发文档时**：按上述「概念 + 操作 + 验证」结构补充，并对命令/步骤做逐条解释。

## 经验总结与知识沉淀方法

- **写哪里**：通用概念、选型理由、项目级决策、易错与排错要点 → `content/knowledge/`（项目在 knowledge 内自定文件名，如决策类、易错类或按主题分）；具体工具用法、导出流程、编辑器/插件操作与编译步骤 → `content/dev/`（可按子目录组织）。同一件事可同时在 knowledge 留决策/易错摘要，在 dev 留操作与配置细节。
- **何时沉淀**：完成一个小阶段或解决一个典型问题后，将「为什么这样选」「踩了什么坑」「具体怎么操作」拆开，分别落入 knowledge 或 dev；用户说「总结经验」时按下一节的流程执行并建议写回。
- **复盘自动化工作流**（取证、门禁、Git 原子提交、pitfalls-inbox 晋升）：除通用沉淀外，可按技能 **retro-automation-workflow** 与 `content/knowledge/15-retro-automation-workflow.md` 的结构化清单执行，并与 **summarize-to-knowledge** 衔接索引更新。
- **与 plan 联动**：阶段/任务完成时，在 plan 的迭代记录中简述结果，并把可复用的收获写入 knowledge 或 dev，保持 plan 简洁、知识可检索。

## 工作区使用 Kit 的方式

- **工作区**：Kit 根 + 项目根（双根）。文档与知识库仅在 Kit 的 `content/` 下；项目根仅放代码与资源。
- **查概念 / 决策 / 易错**：@ `content/knowledge/` 或其下与当前上下文相关的文件；做方案或排错时优先引用此处。具体文件名与结构由 content 内 README 或项目约定说明。
- **查实现与工具用法**：@ `content/dev/` 或其子目录；需要「怎么做、用什么命令、目录与配置约定」时引用 dev。
- **查 API / 示例 / 结构**：@ `content/reference/`；写代码或脚本时引用。reference 下具体子目录与用途由 content 内 README 说明。
- **推进任务与阶段**：读 `content/plan.md`，必要时结合 `content/design/`、`content/dev/` 给出下一步或更新状态。

## 与 AI 协作约定

当用户或对话涉及以下意图时，按下列方式处理（所有文档路径均指**本 Kit 根下的 content/**；代码与资源在**项目根**）：

- **「总结经验」**：根据**当前对话与项目上下文**，自动执行「方法论回顾 + 知识沉淀回顾」：
  1. **方法论回顾**：读取本 Kit 的 `METHODOLOGY.md`，结合当前阶段/任务，简要复述与本轮工作相关的设计-测试-验证-迭代、文档结构、knowledge 与 dev 的边界、编写约定；
  2. **知识沉淀回顾**：读取 `content/knowledge/` 与 `content/dev/` 下与当前上下文相关的条目（以及可选 `content/plan.md`、`content/design/` 中近期更新），归纳已沉淀的要点与可补充的收获；
  3. 基于以上，给出简短「经验小结」与 1–3 条可写回 **knowledge**（概念/决策/易错）或 **dev**（工具用法/导出流程/操作步骤）或迭代记录的建议（在用户同意时执行写入）。
- **长时间未总结经验时的主动提醒**：若对话或会话中已进行多轮开发/设计/调试且**未曾触发「总结经验」**，AI 应主动询问：「是否要总结一下本阶段的经验并沉淀到知识库？」；提醒频率由上下文判断（例如完成一个阶段、或连续多轮实现后），避免打断过频。
- **「更新迭代计划」「生成阶段小结」「根据 plan 给建议」**：读取 Kit 的 `content/plan.md`，可选结合 `content/design/`、`content/dev/`，给出下一步或简短小结，并可在用户同意下建议更新 plan 或设计/开发文档。
- **「阶段 x 完成」「任务 xx 已验证」「本迭代收尾」**：将对应阶段/任务在 `content/plan.md` 中标为已完成或更新最近进展、验证结果与迭代记录；如有可沉淀的收获：概念/决策/易错写入 `content/knowledge/`，工具用法/导出流程/操作步骤写入 `content/dev/`；视情况更新 design。
- **「当前建议推进的任务」或「本阶段下一步」**：读取 `content/plan.md` 与 `content/design/`、`content/dev/`，按阶段与未完成任务给出当前建议推进的任务或下一步。
- **「新增任务」或「拆解阶段」**：在 plan 中新增任务或阶段，更新一览表与详情，并视情况同步 design、dev、knowledge。
- 回答与项目相关问题时：可引用 `content/knowledge/`（概念/决策/易错）、`content/dev/`（实现与工具用法）、`content/reference/`（API/示例/schema）下与上下文相关的文件，并建议把新收获写入 knowledge、dev 或 reference；代码与实现从**项目根**引用。具体要引用的文件由 content 内结构及 README 决定，本方法论不假定 content 内具体路径或文件名。

## 生成阶段小结时的提示（给 AI）

当用户请求「生成阶段小结」或「根据 plan 给建议」时，可：

1. 读取 Kit 的 `content/plan.md`，识别「状态」为「进行中」或最近更新的阶段。
2. 可选：读取 `content/design/`、`content/dev/` 中相关文档，结合任务拆解给出当前建议推进的任务。
3. 根据「验证结果」「迭代记录」与「下一步」生成简短小结与 1–3 条具体下一步行动。
4. 若某阶段长期未更新，可提醒是否要推进或调整。
5. 建议将本次对话中的收获或新问题，写回「迭代记录」或「最近进展」（在用户同意时）。
